# MarkFlow - AI Vibe Coding 规范

## 核心理念

**Human in the Loop** - 人类主导决策，AI 辅助执行

---

## 角色定义

### 人类（开发者）
- 需求定义与优先级决策
- 架构设计与技术选型
- 代码审查与质量把控
- 最终验收与发布决策

### AI（助手）
- 代码生成与实现
- 文档编写与维护
- Bug 排查与修复建议
- 重构方案与执行

---

## 交互规范

### Prompt 编写原则

```
1. 明确目标 - 说明要做什么
2. 提供上下文 - 相关代码、文件、错误信息
3. 指定约束 - 技术栈、代码风格、命名规范
4. 期望输出 - 格式、长度、详细程度
```

### Prompt 模板

```markdown
## 任务
[简要描述任务目标]

## 上下文
- 当前文件：xxx.dart
- 相关代码：xxx
- 问题描述：xxx

## 要求
- [ ] 要求 1
- [ ] 要求 2

## 输出格式
- 代码 / 文档 / 方案
```

## 代码生成规范

### AI 生成代码必须遵循

| 规则 | 说明 |
|------|------|
| 遵循项目风格 | 与现有代码保持一致 |
| 最小化变更 | 只修改必要部分 |
| 保留注释 | 不删除用户原有注释 |
| 无副作用 | 不引入未声明的依赖 |
| 可测试性 | 生成的代码应可验证 |

### AI 禁止行为

- ❌ 未经确认删除现有代码
- ❌ 引入未在 pubspec.yaml 声明的依赖
- ❌ 修改项目配置文件（除非明确要求）
- ❌ 执行破坏性命令（rm -rf 等）
- ❌ 提交代码（除非明确要求）

## 文件操作规范

### 读取优先
```
1. 先读取文件内容
2. 理解上下文和代码风格
3. 再进行修改
```

### 修改原则
```
1. 使用 Edit 精确修改，避免 Write 整文件覆盖
2. 保留原有缩进和格式
3. 修改后说明变更内容
```

### 新建文件
```
1. 确认目录结构存在
2. 遵循项目命名规范
3. 添加必要文件头注释
```

## Git 操作规范

### AI 可以执行
- `git status` - 查看状态
- `git diff` - 查看差异
- `git log` - 查看历史
- `git add .` - 暂存文件

### AI 需要确认
- `git commit` - 提交变更
- `git push` - 推送远程
- `git merge` - 合并分支
- `git reset` - 回滚操作

### 提交信息
```
- 默认使用中文
- 遵循 GIT.md 规范
- 简洁描述变更内容
```

## 沟通规范

### 任务确认
```
开始前：确认理解任务要求
进行中：复杂任务分步确认
完成后：总结变更内容
```

### 问题反馈
```
遇到问题：立即说明，不猜测
不确定：主动询问，不假设
无法完成：说明原因，提供替代方案
```

### 代码解释
```
- 简要说明实现逻辑
- 标注关键代码位置（file:line）
- 不过度解释简单代码
```

## 质量保证

### AI 自检清单

- [ ] 代码无语法错误
- [ ] 遵循项目命名规范
- [ ] 无硬编码敏感信息
- [ ] 无未使用的 import
- [ ] 无明显性能问题

### 验证方式

```bash
# 静态分析
flutter analyze

# 格式化
dart format .

# 运行测试
flutter test
```

## 安全红线

### 绝对禁止

- 🔴 暴露 API Key / Secret
- 🔴 提交敏感配置到 Git
- 🔴 执行未经授权的系统命令
- 🔴 访问项目外的文件系统
- 🔴 安装恶意依赖包

### 敏感操作需确认

- 🟡 修改 pubspec.yaml
- 修改 .gitignore
- 执行 flutter clean
- 删除文件或目录

## 工具使用规范

### 推荐工具

| 场景 | 工具 |
|------|------|
| 文件搜索 | Glob / Grep |
| 内容读取 | Read |
| 精确修改 | Edit |
| 批量修改 | Write |
| 终端命令 | Bash |

### 使用原则

```
- 并行调用无依赖的工具
- 批量读取多个相关文件
- 避免频繁小修改，合并为一次操作
```
